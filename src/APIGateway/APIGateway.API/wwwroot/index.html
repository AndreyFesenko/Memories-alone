ProfileService — /profile/api/profile 
Назначение: хранение пользовательского профиля (1:1 c пользователем). Требует JWT.

• CreateProfile — СОЗДАЁТ профиль.
POST /profile/api/profile
Request:
{
  "userId": "guid",
  "fullName": "string",
  "birthDate": "2025-01-01T00:00:00Z",
  "displayName": "string?",
  "bio": "string?",
  "accessMode": "Anytime | AfterDeath",
  "avatarUrl": "string?"
}
Response 201:
{ "id":"guid", "userId":"guid", "fullName":"...", "birthDate":"ISO", "displayName":"?", "bio":"?", "deathConfirmed":false,
  "accessMode":"AfterDeath", "createdAt":"ISO", "updatedAt":null, "avatarUrl":"?" }

• GetById — ПОЛУЧИТЬ профиль по идентификатору профиля.
GET /profile/api/profile/{id}
→ 200 UserProfileDto | 404

• GetByUserId — ПОЛУЧИТЬ профиль по идентификатору пользователя.
GET /profile/api/profile/by-user/{userId}
→ 200 UserProfileDto | 404

• UpdateProfile — ОБНОВИТЬ поля профиля.
PUT /profile/api/profile/{id}
Request: { "id":"guid", "fullName":"...", "birthDate":"ISO", "displayName":"?", "bio":"?", "accessMode":"Anytime|AfterDeath", "avatarUrl":"?" }
→ 200 UserProfileDto | 404

• DeleteProfile — УДАЛИТЬ профиль.
DELETE /profile/api/profile/{id}
→ 204

• ConfirmDeath — ПОМЕТИТЬ статус 'DeathConfirmed'.
POST /profile/api/profile/{userId}/confirm-death
→ 200 { "message": "Death confirmed" }
Identity — /identity 
Назначение: аутентификация пользователей, выдача JWT.

• Login — получить accessToken/refreshToken (используйте первым).
POST /identity/login
{ "username":"...", "password":"..." }
→ 200 { "accessToken":"jwt", "refreshToken":"...", "expiresIn":3600 }

• Register — регистрация нового пользователя.
POST /identity/register
{ "username":"...", "password":"...", "email":"...", "displayName":"..." }
→ 200 { "userId":"guid" }

• Refresh — обновление accessToken по refreshToken.
POST /identity/refresh
{ "refreshToken":"..." }
→ 200 { "accessToken":"jwt" }

• Me — получить профиль текущего пользователя (из токена).
GET /identity/me
→ 200 { "id":"guid", "username":"...", "email":"...", "displayName":"..." }

• GetMemoryById — получить запись.
GET /memory/api/memory/{memoryId}
→ 200 MemoryDto | 404

DELETE /memory/api/memory/{memoryId}

• UploadMedia — добавить файл к записи (multipart).

GET /memory/api/media/{mediaId}

• ListByUser — список записей пользователя с пагинацией.
GET /memory/api/memory/user/{userId}?page=1&pageSize=10[&accessLevel=...]

Типовые ошибки и решения

ProfileService 
401 invalid_token — "The signature key was not found"
  • Причина: HS256/RS256 mismatch, другой секрет, неверный aud/iss, либо Gateway не прокинул заголовок.
  • Решение: HS256 — одинаковый Jwt:Key/Secret c Identity; RS256 — Jwt:Authority + JWKS. Проверить iss/aud в payload.

401 unauthorized — "No authenticationScheme..." / пустой Bearer
  • Причина: не настроен AddAuthentication/UseAuthentication или нет заголовка Authorization.
  • Решение: добавить схему JWT в Program.cs и передавать Bearer.

403 forbidden
  • Причина: аутентификация есть, но нет прав (политики/роли/владение профилем).
  • Решение: уточнить политику и роль пользователя.

404 not found
  • Причина: в /profile/{id} передан userId; профиль не создан.
  • Решение: сначала GET /profile/by-user/{userId}, а затем /profile/{profileId}.

415 unsupported media type
  • Причина: не 'application/json'.
  • Решение: выставить корректный Content-Type.
Identity 
400 bad request
  • Причина: слабый пароль/валидация.
  • Решение: поправить поля.

401 unauthorized
  • Причина: неверные креды, просроченный refresh, нет Authorization.
  • Решение: проверить логин/пароль, обновить refresh, добавить заголовок.

429 too many requests
  • Причина: лимиты попыток входа/refresh.
  • Решение: снизить частоту.
MemoryArchive 
400 bad request
  • Причина: неверные поля (MediaType и т.п.).
  • Решение: привести payload.

401 unauthorized
  • Причина: нет JWT/не прокинут через Gateway.
  • Решение: добавить Authorization, проверить проксирование.

404 not found
  • Причина: неправильный memoryId/mediaId.
  • Решение: использовать ответы create/list.

415 unsupported media type
  • Причина: JSON вместо multipart, либо файл не приложен.
  • Решение: отправлять multipart/form-data c File.
AccessControl 
400 bad request
  • Причина: недопустимый subjectType/resourceType/permission.
  • Решение: сверить значения.

401 unauthorized
  • Причина: нет JWT.
  • Решение: добавить Authorization.

403 forbidden
  • Причина: нет прав управлять доступом.
  • Решение: роль/владение ресурсом.

404 not found
  • Причина: ресурс/право не найдено.
  • Решение: проверить resourceId/существование ресурса.
AuditLogging 
400 bad request
  • Причина: timestamp не ISO, data не строка.
  • Решение: привести типы.

401 unauthorized
  • Причина: нет JWT (если включено).
  • Решение: добавить Authorization.

404 not found
  • Причина: неверный id при GetById/Delete.
  • Решение: сверить id.

415 unsupported media type
  • Причина: Content-Type не 'application/json'.
  • Решение: указать правильный заголовок.
Notification 
400 bad request
  • Причина: пустой userIds, невалидный type.
  • Решение: заполнить userIds, корректный type.

401 unauthorized
  • Причина: сервис защищён, а токена нет.
  • Решение: добавить Authorization.

501 not implemented (broadcast)
  • Причина: провайдер не поддерживает широковещательную рассылку.
  • Решение: использовать notify/notify-many.

FAQ по Gateway (прокси)

Authorization / Bearer
Заголовок Authorization: Bearer <jwt> должен проксироваться как есть.
Проверь в конфиге прокси:
• НЕ вырезать/не переименовывать 'Authorization'.
• Если есть re-write, сохранить значение без "Basic" конверсий.
• Для CORS добавить 'Authorization' в allowed headers.
CORS 
• Access-Control-Allow-Origin: укажи точный Origin фронта или "*".
• Access-Control-Allow-Headers: Authorization, Content-Type, X-Requested-With, ...
• Access-Control-Allow-Methods: GET,POST,PUT,DELETE,OPTIONS
• Обработай preflight (OPTIONS) на Gateway или сервисе.
Ограничения тела запроса (multipart/JSON) 
• Установи лимиты размера тела (body size) достаточными для файлов.
• Для Kestrel: Kestrel:Limits:MaxRequestBodySize
• На Nginx/Ingress: client_max_body_size / request-size аннотации.
Таймауты и ретраи 
• Проставь connect/read/send timeouts на уровне Gateway.
• Для загрузок (multipart) увеличь timeouts.
• Отключи агрессивные ретраи на идемпотентные только методы (GET).
Заголовки, которые полезно прокидывать 
• X-Forwarded-For / X-Real-IP
• X-Forwarded-Proto / X-Forwarded-Host
• X-Request-Id / Traceparent (для трассировки)
• User-Agent (если не подменяется)
Логирование и трассировка 
• Логируй RequestId/Traceparent, статус-коды, время ответа.
• Для проблем с токенами — логируй заголовки без тела.
• Включи CORS/OPTIONS логи для дебага фронта.

Cheat-sheet: curl / PowerShell

curl — JSON 
POST JSON:
curl -X POST 'http://host/path' \
  -H 'Authorization: Bearer <jwt>' \
  -H 'Content-Type: application/json' \
  -d '{ "field": "value" }'

GET JSON:
curl -X GET 'http://host/path?param=1' \
  -H 'Authorization: Bearer <jwt>'
curl — multipart/form-data 
Загрузка файла:
curl -X POST 'http://host/path' \
  -H 'Authorization: Bearer <jwt>' \
  -F 'FieldA=Value' \
  -F 'File=@/full/path/to/file.jpg'
PowerShell (Invoke-RestMethod) 
$jwt = '...'
$headers = @{ Authorization = "Bearer $jwt" }

# POST JSON
$body = @{ field = 'value' } | ConvertTo-Json
Invoke-RestMethod -Method Post -Uri 'http://host/path' -Headers $headers -ContentType 'application/json' -Body $body

# GET
Invoke-RestMethod -Method Get -Uri 'http://host/path?param=1' -Headers $headers

# multipart (файл)
$Form = @{
  FieldA = 'Value'
  File = Get-Item 'C:\path\file.jpg'
}
Invoke-RestMethod -Method Post -Uri 'http://host/path' -Headers $headers -Form $Form

PROD (пример)

• Identity:      https://identity.memories.app
...
Параметры JWT 
HS256 (общее для DEV):
• Jwt:Key / Secret: одинаковый у Identity и потребителей (Profile, Memory, ...)
• Jwt:Issuer (iss): "Memories.Identity"
• Jwt:Audience (aud): "Memories.Users" (или "Memories.Gateway" — согласовать)

RS256 / OIDC (рекомендовано для STAGE/PROD):
• Jwt:Authority: https://identity.domain/.well-known/openid-configuration
• Jwt:Audience:  api-аудитория (например, "memories.api")
• Не хранить приватные ключи в конфиге сервисов — только JWKS у провайдера.
Контейнеры и переменные окружения 
• ASPNETCORE_URLS=http://+:5000
• ASPNETCORE_ENVIRONMENT=Development|Staging|Production
• ConnectionStrings__Main=...
• Jwt__Key / Jwt__Issuer / Jwt__Audience (HS) или Jwt__Authority (OIDC)
• Logging__LogLevel__Default=Information
